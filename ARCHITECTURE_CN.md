# 系统架构

> **TradingMapClaw（TMC）v2.0 | 2026-07-12**
> 本文档描述系统的设计原则、数据流、组件职责、关键决策与已知局限。

---

## 1. 设计原则

### 1.1 预算纪律

每一项设计决策都要经过每月 **55 美元（¥400 人民币）**硬性预算上限的过滤。这一约束塑造了整个架构：

- 无云基础设施——系统运行在单台 Mac mini（Apple Silicon，macOS 26.5.1）上。
- 无高级付费数据源（Bloomberg、Refinitiv）。数据来自 12+ 免费或低成本来源。
- LLM 成本由自建预算看门狗监控。模型回退链（GLM-5.2 → GPT-5.6 → DeepSeek V4 Pro → Qwen3 14B 本地）从最先尝试的最便宜模型，级联到最后才用的最贵模型。
- 本地模型（通过 Ollama 运行的 Qwen3 14B）负责格式化、去重和免费投票等不需要前沿推理能力的任务，以节省 API 预算。
- 实际测算支出约为每月 7 美元（预算上限的 13.5%），详见 [README 成本明细](README_CN.md#月度成本)。

### 1.2 双引擎、多模型协作委员会

核心分析架构并非单一模型，也不是简单的制作者-核验者两方模式——它是**运行多模型协作委员会的双引擎架构**：

- **双引擎** = 两个相互交叉核验的独立 AI 引擎——**引擎 1：Hermes Agent**（编排者）与**引擎 2：Codex**（独立核验者）——跨三个分析批次协作：
  - **Pass A——引擎 1 / Hermes Agent**（Nous Research，GLM-5.2）：编排 + 基本面、估值、内部人推理。
  - **Pass B——引擎 2 / Codex（GPT-5.6）**：独立的技术面/资金流分析，以及至关重要的——对 Pass A 数字的强制交叉核验。
  - **Pass C——引擎 1 / Hermes Agent**（Nous Research，GLM-5.2）：宏观、行业、情绪、监管推理。
- **多模型协作委员会** = 两个引擎在三个分析批次中调用的模型：**DeepSeek V4 Pro**、**GLM-5.2**、**GPT-5.6**（外加本地 **Qwen3 14B** 用于免费投票）。
- 委员会的角色划分与权重是 **Pass A / B / C**：
  - **Pass A**（Hermes，GLM-5.2）—— 基本面、估值、内部人动向（40%）。
  - **Pass B**（Codex，GPT-5.6）—— 技术面、资金流、期权，以及对 Pass A 数字的强制交叉核验（35%）。
  - **Pass C**（Hermes，GLM-5.2）—— 宏观、行业、情绪、监管背景（25%）。
  - **委员会作战室决胜**（有争议的判断）：DeepSeek → GLM-5.2 → GPT-5.6。

一个模型可能自信地犯错。两个引擎能抓住它。三方组成的委员会来做最终决定。这是系统中最具防御性的单一特性——完整执行流程见[第 3 节](#3-双引擎多模型分析流程)。

### 1.3 零人工干预自动化

系统设计为在交易时段内无需人工干预：

- 118 个定时任务（约 110 个已启用）负责数据采集、分析、报告生成和交付。
- 报告通过 `bilingual_send.py`（一个包裹在冻结模块 `send_reports.py` 外层的直接替换封装）推送到 Telegram 和飞书。
- 质量关卡在报告生成前验证数据的新鲜度和完整性。
- 错误占位符和封装器诊断信息会被列入推送黑名单——用户永远不会收到有问题的报告。
- 系统全天候运行：在美股开盘前（北京时间）采集数据，运行分析，并在交易时段前、中、后向用户手机推送报告。

## 2. 数据流

### 第 1 层：数据采集（输入）

| 方面 | 详情 |
|--------|--------|
| 输入格式 | HTTP API 响应（JSON/HTML）、RSS feed、YAML 配置文件 |
| 采集与分析脚本 | 分布在 `~/.hermes/scripts/` 与 `~/.hermes/cron/scripts/` 中，共 230+ 个 Python 脚本（全部通过编译） |
| 调度 | Hermes 定时引擎——118 个任务，约 110 个启用，约 9 个禁用 |
| 任务构成 | 29 个 LLM 驱动任务 · 76 个纯脚本任务 |
| 代理 | 所有出站流量经 `http://127.0.0.1:10808` |
| 输出格式 | YAML 文件（stocks.yaml、news.yaml、analyst.yaml、sentiment_cache.json、insider_trades.yaml 等） |

数据采集脚本从 12+ 数据源（Yahoo Finance/yfinance、FMP、Finnhub、SEC EDGAR、FRED、Reddit/ApeWisdom、NewsAPI/Alpha Vantage、Polymarket、GDELT、本地 FinBERT、CoinGecko、Binance）拉取数据，将响应归一化为结构化 YAML，写入共享数据目录。

### 第 2 层：质量关卡（预检）

| 方面 | 详情 |
|--------|--------|
| 输入 | 第 1 层的 YAML 文件 |
| 校验 | 数据新鲜度检查、字段完整性检查、schema 校验 |
| API | 端口 8080（程序化访问） |
| 仪表盘 | 端口 8888（可视化监控） |
| 输出 | 传给调度器的通过/失败信号——数据不合格会阻断报告生成 |

质量关卡防止"数据空壳"——即把缺失数据当作完整数据呈现的报告。若必需字段缺失或过期，关卡会中止下游处理，并先触发 4 级回退链（见第 7 节），再考虑标记为不可用。

### 第 3 层：双引擎、多模型协作委员会

| 方面 | 详情 |
|--------|--------|
| 输入 | 经第 2 层校验的数据 |
| Pass A | Hermes，GLM-5.2 —— 基本面、估值、内部人（权重 40%） |
| Pass B | Codex，GPT-5.6 —— 技术面、资金流、期权、对 A 的交叉核验（权重 35%） |
| Pass C | Hermes，GLM-5.2 —— 宏观、行业、情绪、监管（权重 25%） |
| 模型回退链 | GLM-5.2 → GPT-5.6 → DeepSeek V4 Pro → Qwen3 14B（本地，免费） |
| 输出 | 带共识、分歧和置信度标注的结构化分析 |

### 第 4 层：报告生成与双语交付（输出）

| 方面 | 详情 |
|--------|--------|
| 输入 | 第 3 层综合后的分析 |
| 报告类型 | 13 种（T1/T2/T3/T11/T15/T16/T17/T18/T19/T25/T26/R1–R3/可视化） |
| 交付层 | `bilingual_send.py`——直接替换封装，CLI 参数与冻结的 `send_reports.py` 完全相同 |

| 输出格式 | 带纯文本前置元数据的 Markdown 报告——报告正文中出现 YAML 块视为代码缺陷 |
| 交付目标 | 零漏推 |

## 3. 双引擎、多模型分析流程

### 3.1 为什么需要多引擎？

单一 LLM 可能写出一份读起来很像样、但数字错误的分析。TMC 将分析拆分到三个各司其职的独立引擎，并强制 Pass B 在最终报告联合撰写之前核验 Pass A 的数字。

### 3.2 引擎分工

| 引擎 | 模型 | 领域 | 权重 | 输出 |
|--------|-------|--------|--------|--------|
| **Pass A** | Hermes，GLM-5.2 | 基本面 + 估值 + 内部人 | 40% | 1-10 分评分、关键指标、DCF 区间、估值结论 |
| **Pass B** | Codex，GPT-5.6 | 技术面 + 资金流 + 期权 | 35% | 1-10 分评分、进出场点位、交叉核验结论 |
| **Pass C** | Hermes，GLM-5.2 | 宏观 + 行业 + 情绪 + 监管 | 25% | 宏观倾向、行业排名、情绪百分位、监管风险 |

### 3.3 执行流程（全部必须完成）

```
第一步：Pass C 预加载
  → 读取此前运行留存的宏观/情绪记忆
  → 若无记忆：回退至 sentiment_cache.json + macro.yaml
  → 输出：宏观背景文本

第二步：Pass A（基本面）
  → 接收 Pass C 的背景信息 + 全部市场数据
  → 分析：营收增长、利润率、ROE/ROIC、市盈率/远期市盈率/PEG、DCF、负债、内部人
  → 输出：基本面评分 + 关键指标 + 估值区间
  → 若失败：中止（不允许输出部分报告）

第三步：Pass B（技术面 + 交叉核验）
  → 接收 Pass A 的输出
  → 必须核验 Pass A 的关键数字（市盈率、营收增长、目标价、52 周高低点）
  → 若任一数字偏差超过 5%：[交叉核验] 状态标记为 FAIL
  → 分析：RSI、MACD、布林带、EMA、成交量、机构持仓、分析师目标价、空头持仓、期权 P/C 比
  → 输出：技术面评分 + 进出场点位 + 交叉核验结论
  → 若失败：中止（不允许输出部分报告）

第四步：交叉核验检查
  → 解析 Pass B 的 [交叉核验] 状态
  → 若 PASS：置信度 +0.1（上限 1.0）
  → 若 FAIL：将差异标注进分歧部分

第五步：综合（联合撰写）
  → 接收：Pass A 输出 + Pass B 输出 + Pass C 数据 + 交叉核验结果
  → 联合撰写统一的 8 段式报告（不是三份报告的简单拼接）
  → 识别共识（各方一致的内容）与分歧（各方不一致之处，最多 3 点）
  → 输出：买入/持有/卖出 + 加权评分 + 三情景目标价表 + 仓位建议

第六步：Pass C 记忆写回
  → 从最终报告中提取宏观/情绪相关段落
  → 写入记忆（命名空间 engine_c）供后续运行使用
```

### 3.4 报告结构（8 个部分，不超过 1200 词）

| 部分 | 引擎 | 限制 | 内容 |
|---------|--------|-------|---------|
| 1. 核心结论 | 联合撰写 | ≤80 词 | 结论 + 评级 + 目标区间 |
| 2. 基本面 | Pass A | ≤220 词 | 营收/利润率/ROE/估值/DCF/资产负债表/内部人 → 1-10 分 |
| 3. 技术面 + 资金流 | Pass B | ≤220 词 | 趋势/支撑/阻力/RSI/MACD/布林带/成交量/机构/分析师/空头/期权 → 1-10 分 + 进出场点位 |
| 4. 宏观 + 行业 | Pass C | ≤120 词 | 利率敏感性、汇率影响、行业增长、竞争格局 |
| 5. 情绪 | Pass C | ≤120 词 | 新闻情绪、多空比例、分析师趋势、散户与机构对比 |
| 6. 监管 | Pass C | ≤80 词 | 待决监管事项或"无重大风险" |
| 7. 共识与分歧 | 全部引擎 | ≤150 词 | 各方一致的内容 + 分歧点（最多 3 点，各标注采纳的引擎与理由） |
| 8. 最终结论 | 联合撰写 | —— | 星级评级（1-5 星）+ 加权评分 + 三情景概率表 + 仓位建议 + 催化剂 |

### 3.5 硬性约束

1. **所有引擎必须完成**——不允许部分报告。任一引擎失败 → 中止。
2. **交叉核验为强制项**——Pass B 必须核验 Pass A 的数字。
3. **联合撰写**——最终报告是一篇统一文章，而不是三份报告的拼接。
4. **共识 + 分歧**——第 7 部分必须同时呈现两者。
5. **必须给出评级**——不允许"无建议"。必须输出买入/持有/卖出 + 目标区间。
6. **通俗语言**——术语必须附加解释，例如"PEG（市盈率相对盈利增长比率，越低越便宜）0.83"。
7. **仅使用英文**——所有 prompt 和输出均为英文，飞书通过 DeepSeek 获得中文翻译版本。

### 3.6 交互式 vs 定时任务

- **交互式**（用户请求对某标的进行深度分析）：使用 `deep-analysis-dual-head` 技能。Pass B 通过 `delegate_task` 派发；Pass A/C 直接由 Hermes 运行。所有输出汇总后再生成统一报告。
- **定时任务**（`codex_analyst_wrapper.py`）：四步串行流程——Pass C 预加载 → Pass A（Codex CLI）→ Pass B（Codex CLI，交叉核验）→ 综合（Codex CLI）。全部必须完成的关卡机制。

### 3.7 委员会作战室（有争议的判断）

针对更高风险的仓位决策，另有一套三模型投票协议：**第一轮**（DeepSeek 起草）→ **第二轮**（GLM-5.2 复核）→ **第三轮**（GPT-5.6 决胜，仅在第一、二轮意见不一致时启动）。输出：最终买入/持有/卖出 + 置信度（满分 100）。

## 5. T26 供应链 Alpha 管线（摘要）

T26 通过剥离**8 个行业、49 家供应商**的供应链层级来发现投资标的，采用**五层洋葱剥离法**（底层原材料 → 被忽视的中间层 → 核心设备制造商 → 直接受益者 → 终端品牌）。Alpha 通常出现在第 4 层——那些不生产终端产品但握有关键卡位的公司（例如 AXTI 的磷化铟基板、Entegris 的 CMP/高纯化学品）。

评分采用**8 个维度**（业务动能、供应链张力、市场盲区、关联度、营收加速、估值、技术面、情绪）。分类标准：`alpha_score > 0.65 且 mkt_blindness > 0.70` → CANDIDATE（候选），否则 → WATCH（观察）。输出结果写入 `sc_alpha_{行业}.json`，供 T11（Alpha 雷达）和 T26 洋葱报告共同使用。

## 6. T11 Alpha 雷达（摘要）

T11 扫描**8 个数据源**（ApeWisdom 热度激增、社交观察名单、FMP 筛选器、新闻提及、T26 供应链输出、stocks.yaml 备用数据、分析师数据），发现 B/C 观察名单之外的"新面孔"标的。硬性筛选器会剔除无效/场外/meme 股/仙股/微型市值标的。输出分为**三档**：🔴 高确信度（α≥70）、🟡 新兴（α 50-69）、🟠 投机性激增（α<50 + 社交热度激增）。

## 8. 双语交付

- **Telegram** 通过 `bilingual_send.py` 直接接收英文原文 `.md` 文件（绕过冻结的 `send_reports.py` 中文格式化器）。

## 9. 关键设计决策

### 9.1 为什么用 YAML 存储数据（而非数据库）

TMC 将所有中间数据存储为 YAML 文件，而非关系型数据库：
- 零运维开销——无需维护数据库服务器。
- 人类可读——调试就是打开文件看一眼。
- 原子写入——`yaml.safe_dump` + 临时文件重命名保证一致性。
- 数据集很小（82 个标的）——性能不是瓶颈。

权衡：没有查询语言，没有并发写保护。对单人单机系统而言可以接受。

### 9.2 为什么是双引擎 + 委员会（而非单一 LLM）

单一 LLM 可能写出看似合理但数字错误的分析。双引擎模式强制独立核验：Pass A 产出分析，Pass B 独立核验 Pass A 的数字，核验通过则置信度上升，核验失败则在报告中标注分歧。这会消耗更多 API 调用，但能防止最危险的失败模式：自信地犯错。在此基础上叠加多模型协作委员会（DeepSeek V4 Pro + GLM-5.2 + GPT-5.6），进一步降低单一厂商模型偏差悄悄主导整份报告的风险。

### 9.3 为什么用模型回退链（而非单一模型）

GLM-5.2 → GPT-5.6 → DeepSeek V4 Pro → Qwen3 14B 的级联服务于两个目的：
- **成本控制：** 优先尝试更便宜的模型，仅在失败时才启用回退。
- **韧性：** 若某个服务商中断，系统仍能继续运行。
- **本地回退：** Qwen3 14B 通过 Ollama 在本机运行。即使所有 API 服务商都宕机，格式化和去重仍可正常工作。

### 9.4 为什么需要工程宪法

系统有 230+ 个紧密耦合的 Python 脚本。不受控的重构是存在性风险。这套源自 Karpathy `CLAUDE.md` 的 10 条宪法强制要求改前先读、外科手术式改动、无正当理由不加新依赖，以及修复完成前必须验证。这不是风格指南，而是生存协议。

### 9.5 为什么是 WATCHLIST_ONLY（仅观察名单）

系统刻意不连接任何券商 API，也无法执行交易：
- **监管：** 系统是研究工具，而非注册投资顾问。
- **安全：** 使用 LLM 生成的信号进行自动化交易会带来灾难性风险。
- **专注：** 系统的价值在于分析质量，而非执行速度。

## 10. 冻结模块与工程宪法

### 10.1 冻结模块

以下模块已冻结，未经运营者明确批准不得修改：

- `send_reports.py` —— 推送基础设施（1360 行；由 `bilingual_send.py` 绕行，而非替换）
- `runtime_guard` —— 任务守卫系统
- `scheduler` —— 定时引擎
- `stocks.yaml` schema —— 数据 schema

### 10.2 工程宪法

来源：Karpathy `CLAUDE.md`——2026-06-29 提炼。优先级：最高。

1. **改前先读**——修改前先阅读相关文件和现有模式
2. **先想再写**——用一句话说明你在做什么、关键假设、权衡
3. **简单至上**——只写解决当前问题所需的最小代码，不过度抽象
4. **外科手术式改动（硬性规则）**——最小 diff，只动需要动的部分
5. **验证**——先写会失败的测试，再修复
6. **目标驱动**——开始前定义成功标准；先规划再执行
7. **调试**——先调查，别猜测。读完整报错。一次改一处
8. **依赖（硬性规则）**——加新依赖前先检查标准库/现有库
9. **沟通**——说明改了什么、为什么、哪里不确定
10. **失败模式**——警惕 Kitchen Sink（大杂烩）/ Wrong Abstraction（错误抽象）/ Happy Path（只走顺路）/ Runaway Refactor（失控重构）

## 11. 已知局限与权衡

| 局限 | 详情 | 缓解措施 |
|------|--------|------------|
| yfinance 限流 | Yahoo Finance 经常限流请求，部分脚本尚未接入回退链。 | `data_fallback_chain.py` 已存在，但需要逐脚本集成，列入维护待办。 |
| 调度器资源争用 | 多个定时任务在同一分钟触发，造成 CPU/内存争用。 | 待实现专门的错峰或队列优化。 |
| 无 Level 2 数据 | 系统无法访问实时盘口深度数据。 | 架构性限制——需要付费数据源，暂无计划。 |
| 单点故障 | 单台 Mac mini。硬件故障即整体停机。 | 对个人研究工具而言可接受，备份为文件级。 |
| 无并发写保护 | 采集脚本写入 YAML 文件时没有文件锁。 | 因定时调度器对任务串行执行而可接受。 |
| LLM 依赖 | 系统依赖外部 LLM API。若所有服务商都宕机，仅剩本地 Qwen3 14B 可用。 | 模型回退链提供 3 级回退，本地模型可处理格式化但无法生成分析。 |
| 预算约束 | $55/月上限限制了 LLM 调用量和数据源数量。 | 预算看门狗持续强制执行上限，预算紧张时系统优先保障关键报告。 |
| yfinance 冷缓存 | 代码变更后首次运行可能需要 5-10 分钟预热 7 天 TTL 缓存。 | 可接受——后续运行即为热缓存。 |

## 12. 安全边界

TMC 在严格的安全边界内运行：

- **无券商 API：** 系统无法执行交易，不存储任何券商凭证。
- **无用户数据：** 系统不采集或存储运营者以外任何用户的个人数据。
- **本地存储：** 所有数据文件、配置和凭证均存储在 Mac mini 本地，无云存储。
- **API 密钥隔离：** API 密钥和 token 存储在环境变量和本地配置文件中，从不提交至版本控制。
- **代理边界：** 所有出站流量经单一代理（127.0.0.1:10808），采集脚本无直接互联网访问。
- **Hermes+Codex 交叉审计：** 全部脚本经过扫描，完整安全态势见 [SECURITY_CN.md](SECURITY_CN.md)。
- **工程宪法：** 10 条规则约束所有 AI 辅助的代码修改，防止未经审查的变更。
