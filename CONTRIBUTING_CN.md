# 为 TradingMapClaw 贡献代码

> **v1.6.1 | 2026-07-03**

## 系统状态：活跃开发中，但严格受控

TradingMapClaw（TMC）目前是 **v1.6.1**（2026-07-03），处于日常运行和积极维护中。这不是一个接受任意功能请求的绿地项目——它是一个单人维护、拥有 499 个脚本的生产系统，改动受到刻意的约束。

系统有 499 个紧密耦合的 Python 脚本。不受控的改动会带来存在性风险。v1.1 事故——一个缺失的 `id` 字段导致全部 115 个定时任务崩溃——以及 v1.6 的 Codex 审计——发现了一个悄悄禁用 Engine C 记忆功能的 `HERMES_HOME` 未定义变量 bug——都说明了这种纪律的重要性。完整事故与修复历史见 [CHANGELOG_CN.md](CHANGELOG_CN.md)。

## 接受的贡献类型

| 贡献类型 | 状态 | 提交方式 |
|-------------------|--------|---------------|
| Bug 报告 | 接受 | 使用 bug 报告模板开一个 GitHub Issue |
| 文档改进 | 接受 | 开一个带 `docs` 标签的 GitHub Issue 或 PR |
| 数据源建议 | 接受 | 开一个带 `data-source` 标签的 GitHub Issue |

### Bug 报告

使用 `.github/ISSUE_TEMPLATE/bug_report.md` 模板。请包含：
- 发生了什么（观察到的行为）
- 应该发生什么（预期行为）
- 受影响的报告类型/脚本/定时任务
- 问题是可复现还是间歇性的
- 任何错误信息或日志输出

### 文档

以下情况欢迎提交文档 PR：
- 修复现有文档中的事实错误
- 为现有章节增加清晰度
- 改进翻译

以下情况文档 PR 会被拒绝：
- 以文档为幌子添加新功能
- 改变现有技术描述的含义
- 在报告格式描述中引入 YAML 块（视为代码缺陷）

### 数据源建议

如果你知道一个可以替代或补充现有 12+ 数据源之一的免费或低成本数据源，请开一个 Issue，包含：
- 数据源名称和 URL
- 提供什么数据
- 是否有免费额度
- 是否有 Python 库
- 相比现有数据源的优势

注：系统的月度预算上限是 **55 美元**。付费数据源的建议只有在预算范围内才有意义——参见 [ARCHITECTURE_CN.md](ARCHITECTURE_CN.md#1-设计原则) 中的预算纪律原则。

## 不接受的贡献类型

- **随意的功能请求**——新报告类型、新分析模块、新交付渠道都需要有明确的、针对真实缺口的理由，而不是"这样会不错"。门槛之所以高，是因为每增加一个新表面都会加重由一人承担的 499 脚本维护负担。
- **代码重构 PR**——参见工程宪法第 4 条（外科手术式改动）。不受控的重构被归类为"失控重构"失败模式。
- **新增依赖**——参见工程宪法第 8 条。任何新依赖都必须相对标准库/现有库给出正当理由。未经论证就添加 `requirements.txt` 条目的 PR 会被拒绝。
- **修改冻结模块**——`send_reports.py`、`runtime_guard`、`scheduler` 以及 `stocks.yaml` schema 不能被修改。见 [ARCHITECTURE_CN.md](ARCHITECTURE_CN.md#10-冻结模块与工程宪法)。

## 如何提出改动建议

1. **先开 Issue。** 描述问题，而不是解决方案。说明受影响的脚本、定时任务或报告类型。
2. **等待冻结模块检查。** 若提案涉及 `send_reports.py`、`runtime_guard`、`scheduler` 或 `stocks.yaml` schema，会被直接拒绝——请改为提出封装方案（参考 `bilingual_send.py` 的先例）。
3. **提出尽可能小的修复方案。** 按照下方第 4 条规则，PR 应是解决所述问题的最小 diff。
4. **包含验证步骤。** 你如何确认修复有效？`py_compile`、空跑测试、具体测试用例——见第 5 条规则。

## 硬性约束：工程宪法

所有贡献——即使是 bug 修复——都必须遵守工程宪法（来源：Karpathy 的 `CLAUDE.md`，2026-06-29 提炼；见 `skills/engineering_constitution.md`）。10 条规则：

1. **改前先读**——修改代码前先理解现有模式。
2. **先想再写**——动手实现前先说明你在做什么、为什么。
3. **简单至上**——只写解决问题所需的最小代码。
4. **外科手术式改动**——最小 diff。不做顺带的格式改动，不重构无关代码。这是硬性规则。
5. **验证**——修 bug 前先写会失败的测试。
6. **目标驱动**——开始前先定义成功标准。
7. **调试**——先调查别猜测。读完整报错。一次改一处。
8. **依赖**——无正当理由不加新依赖。先问：标准库能否做到？
9. **沟通**——说明你改了什么、为什么、哪里不确定。
10. **警惕失败模式**——若发现自己在做 Kitchen Sink（大杂烩）、Wrong Abstraction（错误抽象）、Happy Path（只走顺路）或 Runaway Refactor（失控重构），立刻停止。

违反这些规则的 PR 会被拒绝，即使代码在功能上是正确的。

## 冻结模块政策

以下模块已冻结，任何 PR 都不得涉及，无论改动多小：

| 模块 | 为什么冻结 |
|--------|------------------|
| `send_reports.py`（1360 行） | 核心推送基础设施。`bilingual_send.py` 是在不动这个文件的前提下扩展交付行为的官方方式。 |
| `runtime_guard` | 系统健康监控——这里的 bug 可能掩盖整条管线中的故障。 |
| `scheduler` | 定时引擎——v1.1 事故（缺失 `id` 字段导致全部 115 个任务崩溃）就发生在这一层附近。 |
| `stocks.yaml` schema | 每一份下游报告和脚本都依赖这个 schema。悄悄修改 schema 会导致 82 个标的的覆盖体系全面失效。 |

如果你的提案需要修改冻结模块，正确的方式是使用封装（参考 `bilingual_send.py` 封装 `send_reports.py` 的做法）——请先开 Issue 说明为什么封装方案不够用，再提出直接修改。

## 审查流程

1. 所有 PR 由维护者（Mickey Wei）审查。
2. 可能使用 Codex 代码审查来验证改动。
3. 涉及核心调度或数据采集的改动需要双引擎（Hermes + Codex）一致同意。
4. 有争议的改动在达成共识前保持不变。

### 关于响应时间的说明

维护者有身体残疾（右臂臂丛神经撕脱、终身造口），行动和打字速度受限。他用一只能用的手审查贡献。响应时间可能慢于典型开源项目。这是单人维护的系统——没有审查团队。

耐心是被感激的。质量优先于速度。
